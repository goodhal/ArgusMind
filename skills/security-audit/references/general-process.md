# 通用安全审计流程

## 审计方法论

ArgusMind 采用 **结构化审计流程**，结合自动化工具和人工分析，确保审计的全面性和准确性。

## 审计阶段

### 1. 项目信息收集

**目标**: 了解项目结构、技术栈和业务逻辑

**步骤**:
1. 使用 `list_files` 获取项目目录结构
2. 识别主要技术栈（语言、框架、数据库）
3. 分析配置文件（数据库连接、API 密钥、认证配置）
4. 识别关键业务逻辑（用户管理、数据处理、API 接口）

**工具**:
- `list_files`: 目录结构
- `read_file`: 配置文件
- `ripgrep_search`: 关键配置项
- `tokei`: 代码统计

### 2. 制定审计计划

**目标**: 根据技术栈和风险类别制定针对性审计策略

**步骤**:
1. 按语言分类（Python、Java、JavaScript、PHP 等）
2. 按风险类别分类（SQL 注入、命令注入、XSS 等）
3. 识别高风险区域（认证、授权、数据处理）
4. 制定审计优先级

**输出**:
- 语言列表
- 每种语言的风险类别
- 审计优先级排序

### 3. Sink 点发现

**目标**: 找到所有安全敏感操作点（Sink）

**步骤**:
1. 搜索危险函数调用（SQL 执行、命令执行、文件操作等）
2. 标记每个 Sink 点的位置和类型
3. 记录 Sink 点的上下文信息（函数名、文件路径）

**工具**:
- `ripgrep_search`: 危险函数搜索
- `read_file`: Sink 点上下文分析
- `code_search`: 函数调用关系

**输出**:
- Sink 点列表（文件、行号、函数名、类型）

### 4. 调用链分析

**目标**: 从 Sink 点逆向追踪到用户输入点（Source）

**步骤**:
1. 从 Sink 点开始，追踪直接调用者
2. 分析数据流（参数传递、变量赋值）
3. 识别防护措施（sanitizer、validator、认证）
4. 评估防护有效性
5. 继续向上追踪直到入口点

**工具**:
- `code_search`: 函数调用关系
- `class_hierarchy`: 类继承关系
- `read_file`: 函数实现分析
- `gitnexus_context`: 代码上下文（可选）

**输出**:
- 完整调用链（Source → Flow → Sink）
- 防护措施分析
- 可利用性评估

### 5. 漏洞确认

**目标**: 确认漏洞的可利用性和严重性

**步骤**:
1. 验证路径可达性（入口点是否可被触发）
2. 验证 Source 可控性（用户是否能控制输入）
3. 验证防护缺失（防护措施是否有效）
4. 评估漏洞严重性（数据泄露、权限绕过等）

**判定标准**:
- **LIKELY_VULNERABLE**: 路径可达 + Source 可控 + 防护缺失
- **SAFE**: 任一条件不成立

**输出**:
- 漏洞判定结果
- 可利用性分析
- 严重性评估

## 工具使用指南

### ripgrep_search

**用途**: 快速文本搜索，查找危险模式

**示例**:
```python
# 搜索 SQL 查询
ripgrep_search("execute|cursor\.execute|SELECT.*FROM")

# 搜索命令执行
ripgrep_search("system|exec|popen|subprocess")

# 搜索文件操作
ripgrep_search("open|read|write|file_get_contents")
```

### code_search

**用途**: 搜索类、方法、字段定义

**示例**:
```python
# 搜索类定义
code_search("class UserController")

# 搜索方法定义
code_search("def execute_query")

# 搜索字段使用
code_search("user_id")
```

### read_file

**用途**: 读取文件内容，分析具体实现

**示例**:
```python
# 读取完整文件
read_file("src/api/users.py")

# 读取特定行范围
read_lines("src/api/users.py", 100, 150)
```

### class_hierarchy

**用途**: 查找类的继承关系

**示例**:
```python
# 查找父类
class_hierarchy("UserController", "parents")

# 查找子类
class_hierarchy("BaseController", "children")
```

## 数据流分析

### Source（输入点）

**常见 Source**:
- HTTP 参数（GET、POST、Header）
- 文件上传
- 数据库查询结果（跨媒介）
- 配置文件（跨媒介）
- 环境变量

**验证方法**:
1. 检查参数来源（HTTP request 对象）
2. 确认用户可控性（是否有写入接口）
3. 分析数据类型和格式

### Flow（传播路径）

**常见 Flow**:
- 函数调用（参数传递）
- 变量赋值
- 数据转换（编码、解码）
- 条件分支

**分析方法**:
1. 追踪变量使用
2. 分析数据转换
3. 检查条件分支影响

### Sink（敏感操作）

**常见 Sink**:
- SQL 执行
- 命令执行
- 文件操作
- 网络请求
- 模板渲染
- 反序列化

**分析方法**:
1. 确认操作类型
2. 检查参数来源
3. 评估操作风险

## 防护措施分析

### Sanitizer

**定义**: 清理或转义危险字符的函数

**验证方法**:
1. 检查 sanitizer 实现是否正确
2. 确认是否覆盖所有危险字符
3. 测试是否可绕过

**常见问题**:
- 只过滤部分危险字符
- 过滤顺序错误
- 编码绕过

### Validator

**定义**: 验证输入是否符合预期格式

**验证方法**:
1. 检查验证规则是否严格
2. 确认是否覆盖所有输入
3. 测试边界条件

**常见问题**:
- 验证规则不严格
- 类型混淆
- 边界条件处理不当

### Authz（授权）

**定义**: 检查用户是否有权限执行操作

**验证方法**:
1. 检查授权逻辑是否正确
2. 确认是否覆盖所有操作
3. 测试权限绕过

**常见问题**:
- 授权检查缺失
- 权限判断错误
- IDOR（不安全的直接对象引用）

## 最佳实践

### 1. 系统化分析

- 不要跳过任何阶段
- 使用工具验证假设
- 记录所有发现

### 2. 证据驱动

- 每个结论都要有代码证据
- 不要猜测或假设
- 提供具体的代码位置

### 3. 上下文理解

- 理解业务逻辑
- 分析完整调用链
- 考虑防护措施

### 4. 严重性评估

- 基于可利用性评估
- 考虑业务影响
- 区分实际风险和理论风险

## 常见错误

### ❌ 猜测式分析

**错误**: 假设漏洞存在而不验证
**正确**: 使用工具读取代码，确认漏洞

### ❌ 单点分析

**错误**: 只看单个函数，不考虑调用链
**正确**: 分析完整的数据流和调用链

### ❌ 忽略防护

**错误**: 发现危险函数就报告漏洞
**正确**: 检查防护措施的有效性

### ❌ 过度报告

**错误**: 报告所有潜在风险
**正确**: 按严重性和可利用性分类

### ❌ 缺少证据

**错误**: 只说"存在漏洞"
**正确**: 提供具体的代码证据和位置
